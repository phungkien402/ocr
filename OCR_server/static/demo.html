Overview

API trích xuất 7 chỉ số sinh tồn (mạch, nhiệt độ, huyết áp, nhịp thở, cân nặng, chiều cao, SpO₂) từ ảnh thiết bị y tế hoặc phiếu kết quả viết tay. Pipeline lai YOLO (BP monitor, ~0.4s) + Qwen2.5-VL fallback (~3s).

HITL bắt buộc: client KHÔNG được auto-submit response về EMR. Mọi trường có thể null — phải cho bác sĩ xác nhận hoặc gõ tay trước khi ghi hồ sơ.

Endpoints

POST /v1/extract

Endpoint chính. Yêu cầu auth.

GET /health

Public, không auth. Trả status server + VLM circuit breaker state.

GET /test

UI test tối giản cho /v1/extract.

POST /process

Legacy debug endpoint — full internal result (raw VLM output, debug steps). Dev only.

Behavior chi tiết

Pipeline ưu tiên

  1. YOLO fast path (~0.4s): nếu ảnh là BP monitor và detect đủ 3 giá trị SYS/DIA/PUL → trả luôn, không gọi VLM.
  2. VLM fallback (~3s): các ảnh khác đi qua Qwen2.5-VL-3B. Có retry tự động (2 lần) + circuit breaker.

VLM circuit breaker

Khi vLLM crash, sau 5 consecutive failures circuit chuyển OPEN trong 60s — request mới trả null vitals ngay (~10ms) thay vì đợi timeout đầy đủ. State expose qua /health.

Rate limit

300 request / phút mỗi API key. Khi vượt → 429. Client BẮT BUỘC backoff trước khi retry.

Image size

Hard limit 10MB. Mobile client nên compress xuống ≤1MB trước upload (JPEG q85) để giảm latency upload + bandwidth.

Best practices cho PHR client

  1. Setup screen lưu endpoint + api_key vào secure storage 1 lần. Bác sĩ không gõ tay từng request.
  2. Compress ảnh ≤1MB JPEG q85 trước khi upload.
  3. HITL form prefill từ response, trường null để trống cho bác sĩ điền. Nút "Xác nhận" mới push EMR.
  4. Error handling:
    • 401 → check config, không loop retry
    • 429 → backoff 5s, 10s, 20s...
    • 500/network → hiện form trống, không block UI
  5. Lưu request_id cùng record bệnh án để trace ngược về ảnh gốc trên server nếu cần audit.
